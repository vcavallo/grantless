# Test Plan: Story 6.5 — Role-based task management on the detail page

**Story:** `.pi/engineering-team/stories/6.5-task-management-detail-page.md`
**ADR:** `.pi/engineering-team/decisions/0008-task-management-detail-page.md`
**Date:** 2026-06-03

## Approach

Three tiers, matching the project's philosophy:

1. **Unit (`npm test`, no relay):** the transition/conclusion logic is pure (`markTaskStatus`, `assignWorker`, `buildTaskConclusionTemplate`, `buildMockZapReceiptTemplate`, `parsePubkey`), so it's tested directly and round-tripped through the existing parsers (`parseTaskProposal`, `parseTaskConclusion`).
2. **Real-event e2e (`npm run test:e2e`):** `task-lifecycle.e2e.test.ts` drives a task `proposed → funded → in_progress → submitted → concluded` against a live local strfry using the **app builders**, each transition signed by the acting role, asserting each step is authoritative and the 3402 references the task + payout.
3. **Browser (`npm run test:browser`):** `task-management.spec.ts` drives the loop through the real UI by role on seeded tasks, and asserts role-gating (a non-participant sees no actions).

## Test infrastructure

- Unit: Vitest; fixtures via `buildTaskProposalTemplate` → `parseTaskProposal`; reuse `parseTaskConclusion`.
- E2E: Vitest + local strfry + `nak`, reusing `test/e2e/harness.ts`, driving `@/lib/catallax`.
- Browser: Playwright vs the seeded relay (`PLAYWRIGHT_CHROMIUM_PATH` on NixOS, port 8123); `helpers.ts` `loginAs`/`rememberCurator`; seeded tasks at each status drive the role transitions.

## How to run

```
npm test            # unit
npm run test:e2e    # nak full-loop e2e
npm run test:browser  # role-gated UI loop (NixOS: set PLAYWRIGHT_CHROMIUM_PATH)
```
