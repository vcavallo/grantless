# Story 2.5: Browser e2e (Playwright) + UX hardening

**Status:** Approved
**Created:** 2026-06-03
**Type:** Feature (test infrastructure + UX/bug fixes)

## Background
The nak/relay-level e2e suite (Story 2) proves protocol behavior, but the **rendered app** has only ever been checked by hand — and that manual noodling has now missed three real defects in a row: a persisted-session "No curators found" regression (works in a fresh/incognito tab, fails in a returning logged-in session), no visible login button (you can only log in via the "Post a project" dialog), and the arbiter assign/change control living only on the browse index, not on the task's own detail page where you'd manage it. This is exactly the "outer ring" the epic flagged for a Playwright fast-follow (Story 2.5).

This story stands up **browser-level e2e with Playwright** against the seeded local relay, uses it to **reproduce and fix the persisted-session regression**, and closes the two UX gaps (**login affordance**, **arbiter control on the detail page**) — hardening the foundation before funding (Story 7) lands.

## User-facing description
As a Grantless user, I want the app to behave correctly across real browser sessions — discover curators whether or not I've used it before, log in from an obvious button, and manage my project's arbiter from the project's own page — and as a developer, I want browser tests that catch these regressions automatically instead of relying on manual checking.

## Acceptance criteria
- [ ] **Playwright harness.** Given a single documented command, then Playwright launches the app pointed at the **seeded local relay** and runs the browser specs headlessly, with no manual setup beyond what `npm run seed` already provides; teardown leaves nothing running. The fast unit gate (`npm test`) and the nak e2e (`npm run test:e2e`) are unchanged and still pass.
- [ ] **Persisted-session regression fixed + guarded.** Given a returning browser session with pre-existing local state (a prior persisted relay config and/or a logged-in account) against the seeded relay, when the page loads, then the curator picker is populated and the seeded curator's applicants render — there is **no false "No curators found"**. A Playwright spec reproduces the returning-session scenario (seed `localStorage`, reload) and asserts curators render; it fails before the fix and passes after.
- [ ] **Visible login affordance.** Given a logged-out visitor on the browse page, then a clear **login control is present in the page header** (independent of the "Post a project" dialog); activating it opens the login flow; once logged in, the header shows the account/switcher. A Playwright spec asserts the logged-out header shows a login control.
- [ ] **Manage the arbiter from the task detail page.** Given the patron is viewing **their own** project's detail page, then they can assign or change the arbiter there — chosen from curator-vouched options (the `grantless-arbiter` chain, as in Story 6) — and the change persists (latest version wins); a non-patron viewing the page sees no such control. (How the curator context is resolved on a non-curator-scoped page is the Architect's call.)
- [ ] **Core-flow browser coverage.** Playwright specs cover, against the seed: (a) the browse renders the seeded curator → applicants → their projects; (b) logging in with a seed nsec; (c) posting a project and seeing it appear under the author; (d) assigning an arbiter (from the browse and/or the detail page). These encode the flows Stories 4–6 delivered so future changes can't silently break them.
- [ ] **No privileged actor / openness preserved.** The browser tests run against a local relay with seed/throwaway keys; no pubkey, relay, or arbiter is special-cased; the login/arbiter UX adds no privileged default. Existing openness properties (curator-overridable, resolve-by-observer) are unaffected.

## Out of scope
- **Story 7+ feature work** (funding/zap goals, worker assignment, conclusions) — this is hardening, not new protocol features.
- **CI integration / cross-platform** runs — local dev-machine use is sufficient (consistent with Story 2's scope); wiring CI is a later chore.
- **Visual-regression / screenshot** testing — behavioral assertions only.
- **Reworking the legacy `/catallax` dashboard** beyond what the arbiter-on-detail change requires.
- **A full app-wide layout/header redesign** — the login affordance is added where users land (the browse page header), not a sweeping nav overhaul.
- **Deleting the now-unmounted pasted-naddr path** (`NomineeBrowser`/`useNomineeList`) — still a later cleanup.

## Openness check
Pure tooling + UX hardening. The Playwright harness drives a local relay seeded with throwaway keys — no privileged actor, no live-relay dependency. The login button surfaces the existing, key-agnostic login flow. The arbiter-on-detail control reuses Story 6's curator-vouched, observer/source-tag resolution (no new privilege). The regression fix concerns relay-config precedence for a *dev/test convenience* and must not introduce any elevated default. Fork test unaffected: anyone runs the harness against their own seeded relay identically. Consistent with the prime directive.

## Open questions
Resolved during planning (decisions for the autonomous run; flagged for review veto):
- **Browser runner:** Playwright (the epic's named choice), headless Chromium, against `npm run dev`/seed. Architect picks the exact wiring (webServer config, port, seed reuse).
- **Regression root cause:** not yet pinned by inspection (login-tied; works incognito). The Architect reproduces it with Playwright first, root-causes, then fixes — the fix design comes from the ADR, not a guess here.
- **Arbiter context on the detail page:** the detail page isn't curator-scoped. Candidate approaches (Architect decides): reuse the remembered last-selected curator; offer a small curator picker; or fall back to the union of curators that list the patron. Must stay curator-vouched and overridable.
- **Login affordance placement:** the Grantless browse page header (where users land); extending to other pages is optional.
